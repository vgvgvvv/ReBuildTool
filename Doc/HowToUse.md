# How to Use ReBuildTool

ReBuildTool (RBT) is a C#-driven native build system, in the spirit of Unreal
Engine's UBT: you describe **Targets** and **Modules** as small C# classes
(`.target.cs` / `.module.cs`), and RBT compiles those rule files on the fly,
then uses them to drive a real C/C++ toolchain (MSVC, Clang, Gcc, or Wasm) and
to generate an IDE project (Visual Studio / CMake).

This document covers day-to-day usage. For the high-level pitch see
[Whats-ReBuildTool.md](Whats-ReBuildTool.md).

(中文版: [HowToUse.zh-CN.md](HowToUse.zh-CN.md))

## 1. Building RBT itself

RBT ships as source; there's no prebuilt binary in the repo. Build it once
with the scripts in [BuildScript](../BuildScript):

```bash
cd BuildScript
./BuildAll.sh        # or BuildAll.bat on Windows
./BuildUpdater.sh    # or BuildUpdater.bat
```

This runs `dotnet publish` (self-contained, .NET 8) for `ReBuildTool.csproj`
and `ReBuildTool.Updater.csproj` for `win-x64`, `osx-x64`, `osx-arm64`, and
`linux-x64`, dropping binaries into `ReBuildTool/Binary/<OS><Arch>/...`.

After that, `rbt.sh` / `rbt.bat` and `rbt-updater.sh` / `rbt-updater.bat` are
thin dispatchers that just exec the platform binary for the current OS/arch
and forward all arguments, e.g.:

```bash
./BuildScript/rbt.sh --ProjectRoot . --Mode Build --Target MyGame
```

## 2. Bootstrapping a new project with the Booster

You don't need a local RBT checkout to start a new project — copy
`BuildScript/RBTBooster.sh` (or `.bat`) into an empty folder and run:

```bash
./RBTBooster.sh --init [TargetName]
```

What it does:
1. Clones `ReBuildTool` into `$RBT_HOME` (defaults to `~/.rbt`, override with
   the `RBT_HOME` env var) if it isn't there yet, or offers to rebuild it.
2. Builds RBT + the Updater from that clone (`BuildAll`/`BuildUpdater`).
3. Runs the built `ReBuildTool` binary against the current directory in
   `Init` mode, which generates a starter Target/Module and an IDE project.
4. RBT then copies its own `RBTBooster.sh/.bat` next to your project and
   writes two convenience wrapper scripts into the project root:
   `InitProject.sh/.bat` and `BuildProject.sh/.bat` (thin wrappers around
   `RBTBooster --init` / `--build`). **Don't edit these by hand** — they're
   regenerated by the tool.

Subsequent builds:

```bash
./RBTBooster.sh --build [TargetName]
# or, once generated:
./BuildProject.sh
```

## 3. Running the tool directly

Every RBT invocation resolves to one call of the `ReBuildTool` executable
with a `--Mode` and (usually) a `--Target`:

```bash
ReBuildTool --ProjectRoot <path> --Mode <RunMode> --Target <name> [options...]
```

| Flag | Meaning |
|---|---|
| `--ProjectRoot <path>` | Project root folder. Defaults to the current working directory. |
| `--Mode <RunMode>` | **Required.** One of `Init`, `Build`, `Clean`, `ReBuild`. |
| `--Target <name>` | Target to build. Defaults to the `ProjectRoot` folder name. |
| `--RunDry` | Dry run — parse/validate without actually compiling. |
| `--BoosterSource <path>` | Internal; set by the Booster scripts so RBT can regenerate them. Don't set manually. |

`Mode` behavior (see [Program.cs](../ReBuildTool/ReBuildTool/Program.cs)):
- **Init** — scaffolds a default Target/Module under `Source/` if none
  exist, then generates the IDE project (`.sln` for Visual Studio, or CMake).
- **Build** — compiles the given target.
- **Clean** — removes build outputs.
- **ReBuild** — `Clean` followed by `Build`.

RBT actually processes **two parallel project types** on every run: a
C#-rule-based C++ project (`ICppProject`, described below) and a legacy
INI-based project (`IIniProject`, see §6). Both are parsed and dispatched
through the same `Mode`.

### C++-specific flags

These are only relevant when building native (C/C++) targets:

| Flag | Meaning |
|---|---|
| `--TargetPlatform` | e.g. `Windows`, `Linux`, `MacOSX`, `iOS`, `Android`, `Wasm`. |
| `--TargetArch` | `x86` \| `x64` \| `arm32` \| `arm64`. |
| `--BuildConfig` | `Debug` \| `Release` \| `ReleasePlus` \| `ReleaseSize`. |
| `--UseClang` | Use Clang instead of the platform's default toolchain (MSVC on Windows, Gcc/Clang on Linux). |
| `--ClangPath` | Explicit path to a Clang install. |
| `--CustomIncludeDirs`, `--CustomDefines`, `--CustomCompileFlags`, `--CustomLinkFlags`, `--CustomArchiveFlags` | Extra flags/paths appended on top of what the modules declare. |
| `--CustomStaticLibraries`, `--CustomDynamicLibraries`, `--CustomLibraryDirectories` | Extra libraries/library search paths. |
| `--CppCompilePlugins` | Names of `ITargetCompilePlugin` plugins to run pre/post compile. |
| `--UseMakeFileBuild` | Generate and drive a Makefile build instead of invoking the toolchain directly (default `true`). |
| `--DebugToolchainCmd` | Print the exact toolchain command lines being run. |

Platform-specific groups (only relevant for the matching `--TargetPlatform`):
- Android: `--NDKRoot`, `--SDKRoot`, `--NDKTargetVersion` (default `25`)
- iOS: `--IOSTargetVersion` (default `15.0`)
- macOS: `--MacOSXTargetVersion` (default `11.5`)

Supported native toolchains: **MSVC** (VS2017/2019/2022 auto-detected),
**Clang** (Windows/Linux/macOS/iOS/Android), **Gcc**, and **Wasm**.

## 4. Writing a C++ project — Targets and Modules

Under `<ProjectRoot>/Source/`, RBT discovers any `*.target.cs`,
`*.module.cs`, and `*.extension.cs` files, compiles them together into an
in-memory `CompileRules.dll`, and reflects over the resulting types.

See the worked example in [Sample/BuildCpp](../Sample/BuildCpp):
[Main.target.cs](../Sample/BuildCpp/Source/Main.target.cs) and
[MainModule.module.cs](../Sample/BuildCpp/Source/MainModule/MainModule.module.cs).

> **Note:** the checked-in sample uses the short class names `TargetRule` /
> `ModuleRule`, which no longer exist in the codebase. The tool's own
> scaffolder (used by `--Mode Init`) generates `CppTargetRule` /
> `CppModuleRule` — use those names for anything new.

### Target rule (`*.target.cs`)

```csharp
using ReBuildTool.ToolChain;

public class MyGameTarget : CppTargetRule
{
    public MyGameTarget()
    {
        UsedModules.Add("MyGameModule");
    }
}
```

Key members of `CppTargetRule`:
- `List<string> UsedModules` — modules linked into this target (entry points).
- `List<GitLibrary> GitLibraries` — external git dependencies (`Name`, `Url`, `Branch`).
- `List<ITargetCompilePlugin> Plugins` — pre/post-compile hooks.
- `virtual void Setup(ICppBuildContext)` / `virtual void PostBuild()` — override for custom logic.

### Module rule (`*.module.cs`)

```csharp
using ReBuildTool.ToolChain;

public class MyGameModule : CppModuleRule
{
    public MyGameModule()
    {
        TargetBuildType = BuildType.StaticLibrary; // or DynamicLibrary / Executable
        Dependencies.Add("SomeOtherModule");
        PublicIncludePaths.Add("Public");
    }
}
```

Key members of `CppModuleRule`:
- `BuildType TargetBuildType` — `StaticLibrary` \| `DynamicLibrary` (default) \| `Executable`.
- `List<string> Dependencies` — other modules this one depends on.
- Public/Private pairs, each affecting only this module (`Private...`) or also
  everything that depends on it (`Public...`):
  `IncludePaths`, `Defines`, `CompileFlags`, `LinkFlags`, `ArchiveFlags`,
  `StaticLibraries`, `DynamicLibraries`, `LibraryDirectories`.
- `List<string> SourceDirectories` — extra source dirs beyond the module folder.
- `bool IsSupport` — override to gate a module in/out per platform.
- Per-compilation-unit hooks: `CompileFlagsFor`, `DefinesFor`, `IncludePathsFor(CppCompilationUnit)`.
- Builder hooks: `AdditionCompileArgs`, `AdditionLinkArgs`, `AdditionArchiveArgs`.

`UnityModuleRule` (a `CppModuleRule` subclass) is available for unity/jumbo
builds and auto-generates `<Module>.internal.h/.cpp` import/export macro pairs.

### Expected folder layout

`--Mode Init` scaffolds (and expects) this shape:

```
<ProjectRoot>/
  Source/
    MyGameTarget.target.cs
    Src/
      MyGame/
        MyGameModule.module.cs
        Public/
          MyGameModule.h
        Private/
          MyGameModule.cpp
```

## 5. Programmatic / lifecycle API

Every project (C++ or INI) exposes the same lifecycle, also used internally
by `Program.cs`'s mode dispatch and by the NUnit tests in
[ReBuildTool.Test](../ReBuildTool/ReBuildTool.Test):

```csharp
project.Parse();   // discover & compile *.target.cs / *.module.cs
project.Setup();   // scaffold defaults if needed + generate IDE project
project.Build(targetName);
project.Clean();
project.ReBuild(targetName);
```

## 6. Legacy INI project format

Alongside the C# rules, RBT still parses an INI-based project/module format
(`ReBuildTool.Ini`), run in parallel on every invocation. A generated default
`.target`-style file looks like:

```ini
[Target]
+Entries="Runtime"

[Init]
+DependOn="Action:DoSomething"
+Actions=(Name="ReMake.Init", Args=(projectName="Sample"))

[Build]
# build actions

[Action:DoSomething]
+Actions=...
```

and correspondingly for `.module.ini` files with a `[Module]` section and
`+Dependencies=`. This format predates the C# rule system — prefer
`.target.cs` / `.module.cs` for new projects unless you specifically need
the INI action/target-dependency graph (built on `Bullseye`).

## 7. Self-update

`ReBuildTool.Updater` (invoked via `rbt-updater.sh` / `rbt-updater.bat`) pulls
the latest `ReBuildTool` git repo into `$RBT_HOME` and rebuilds it from
source (`BuildAll` + itself), i.e. RBT bootstraps/updates itself from its own
repository:

```bash
./BuildScript/rbt-updater.sh
```

## 8. Quick reference

```bash
# one-time setup in an empty project folder
./RBTBooster.sh --init MyGame

# rebuild after changing Target/Module rules or source
./BuildProject.sh
# equivalent to:
./RBTBooster.sh --build MyGame

# direct invocation with more control
ReBuildTool --ProjectRoot . --Mode Build --Target MyGame \
    --BuildConfig Release --TargetPlatform Windows --TargetArch x64
```
